# 版本管理非常危险，有必要新开一个章节

## 首先陈述下我遇到的问题

- 真是乐极生悲。我昨天上完组合数学，刚好弄了那个所谓的“太爽了”的范畴论。当时下课了，外面在下大雪，路面上都是冰泥。教室里很冷，肚子还饿，我急着吃饭，想要顺便push一下仓库
- 结果打开git后发现我一直都在“拓扑学修改前的版本”分支下编辑。也就是说，我忘了切换回master分支了。半个月的东西都写错了。直到我add完了后才发现这一点。而我当时急火火的，想着直接切换回master得了，反正各个分支共享工作区和暂存区。但是不断用switch（谢天谢地没有用checkout），它一直警告我让我先stash暂存区。我想着没办法顺应你一下吧，就随便写了个默认的stash。然后commit、push一气呵成，就直接吃饭去了
- 结果今天上午我光折腾抽代了，还没发现拓扑已经完蛋了。直到下午数值线代，瞎逛了下远程仓库，结果发现，欸我拓扑呢。完了完了，仔细一看更不得了了，我这master分支怎么回退到11月27日的版本了？泛函还没写到H-B定理，上周日七个小时总结的拓扑映射部分也全部GG了。这一下真的让我使用git以来最害怕的东西成真了。当时我差点以为15号楼又要多一具尸体了。尤其让自己重新写一遍已经写过的东西，而且在写的过程中总觉得这里应该有东西没写进去，但就是想不起来啥玩意没写。这是最折磨人的。
- 然后我就折腾了整整一下午加一晚上，才最终因为VS Code的时间线功能救了我的老命。

### 最终补救办法

- 首先，因为我之前11月27日左右设置了“拓扑学修改前的版本”分支，所以很多文件夹目录下的文件还和原来一样
  - 那么，用 VS Code 打开它们，在时间线中找出最后的版本（不知为什么，只要是同名文件，那么所有分支就共享一个时间线）。然后将内容复制到新建md中，然后切换回master分支，进行粘贴即可
- 并且 `git stash apply` 也不是一无是处。它把我的昨天晚上写下的所有文件都吐了出来。只需重设这些文件夹目录的结构即可
- 综上，误删的文件用之前存储的分支修复，被重构的文件夹用stash进行修复。这样我就获得了之前所有的文件，真的是一丁点也没有落下

### 吸取的教训

- **一定一定要勤用commit**！每次进行重大修改后一定要commit一下！如果嫌弃commit太多很混乱，可以使用 `git commit --amend` 覆盖最后一次commi，然后推送时用 `git push ... -f` 覆盖远程仓库就行。但是一定要做！文件树复杂一点没关系，重要的是别丢了，否则真的找不回来
- **切换分支前一定要commit一下，stash根本没用**！这次我是以为stash了就可以把东西找回来，没想到这个老逼登只愿意把抽代的部分文件给我吐出来。之前拓扑泛函数理统计和日记全给我扔了！
- **在进行危险的版本管理前，一定要先用网盘或本地文件复制备份一下**
